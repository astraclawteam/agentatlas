# AgentAtlas 企业知识维护体验设计

**日期：** 2026-07-10

**状态：** 已确认

**范围：** AgentAtlas Console、Atlas 助手、AgentNexus 身份与审批边界、`xiaozhiclaw-runtime` 共享 UI 包

## 1. 决策摘要

AgentAtlas 是企业知识库运行时及其知识维护前端。普通员工不知道也不直接操作 AgentAtlas；员工通过安装在 PC 上的小智 Claw，经 AgentNexus 企业网关使用知识。

企业管理者、运维人员和服务商通常也不需要每天打开 AgentAtlas。只有在知识纠错、资料入库、SOP/工作流维护、建议审核或高级诊断时才直接使用 AgentAtlas。

第一版体验采用以下已确认决策：

- 小智 Claw 可以携带任务上下文打开 AgentAtlas，但 AgentAtlas 不嵌入、不启动、也不依赖小智桌面运行时。
- AgentAtlas 常态工作区以组织范围内的知识浏览为主体，吸收任务快捷入口和内容搜索。
- 基础界面由管理者、运维人员和服务商共用；只有具备 `service_mode` 权限的人才能切换高级维护模式。
- 小智带来的简单变更优先在小智内完成快捷确认；需要查看修改前后、影响范围或流程结构时打开 AgentAtlas 专注确认页。
- 无编辑权限的员工可以提交纠错建议，但不能直接修改或发布。
- 低风险修改允许具备权限的用户单人确认发布；高风险修改必须沿组织架构向上进行双人复核，发起人不能复核自己的修改。
- 风险级别由确定性企业策略决定。Agent 只解释判定原因，不能降低风险。
- AgentAtlas 不显示手工票据输入框。已有会话时无感进入；未登录时整页跳转企业登录；禁止登录弹窗和复制 token/ticket。
- AgentAtlas 的聊天、输入、基础组件和视觉系统来自 `xiaozhiclaw-runtime` 中唯一维护的可发布共享包，不复制组件源码。

## 2. 产品边界

### 2.1 员工使用路径

```text
普通员工
  → 小智 Claw
  → AgentNexus（身份、权限、连接器、票据与审计）
  → AgentAtlas（知识、SOP、记忆、检索与证据索引）
  → 企业数据和知识
```

员工日常只在小智 Claw 中提问、执行 SOP、提交工作信息或提出纠错建议。AgentAtlas 的存在不构成员工使用产品时必须理解的概念。

### 2.2 直接使用 AgentAtlas 的人员

- 企业管理者：维护本组织知识、处理建议、审核和发布。
- 企业运维或知识负责人：维护资料、知识、SOP 和组织适用范围。
- 服务商或高级维护人员：在授权范围内使用工作流画布、解析诊断、连接状态、版本和索引维护能力。

### 2.3 明确不做

- 不把 AgentAtlas 变成普通员工门户。
- 不嵌入小智 Claw 桌面应用、原生桥、stores、会话状态或 gateway event bus。
- 不让用户手工填写 Access Ticket、企业 ID、组织 ID、JSON、cron、索引名或节点类型。
- 不把基础界面做成传统多级 SaaS 后台或通用低代码平台。
- 不让 Agent 自行决定权限或降低企业风险级别。

## 3. 体验模型

AgentAtlas 有两种进入方式，但共享同一权限和版本体系。

### 3.1 直接打开

管理者、运维人员或服务商直接打开 AgentAtlas，进入常态知识维护工作区。系统自动恢复其企业身份、可维护组织范围和权限，不要求选择技术租户或粘贴票据。

### 3.2 小智带入

员工在小智中提出“修改这条知识”“把这份文件做成 SOP”等请求后：

1. AgentNexus 验证员工身份、组织范围和建议/编辑权限。
2. AgentAtlas 创建建议或修改草稿。
3. 小智打开一个带签名、限时、任务上下文的 AgentAtlas 链接。
4. AgentAtlas 登录完成后直接进入对应建议、修改或审批任务，不经过常态首页。

小智链接只携带不敏感的任务标识和防篡改上下文；原始证据和权限仍由 AgentNexus 按需验证与读取。

## 4. 信息架构

基础界面保留技术设计规定的四个工作面，但采用普通用户能理解的名称：

| 界面名称 | 技术对象 | 默认能力 |
|---|---|---|
| 企业知识 | Knowledge Space、SOP、资料 | 浏览、搜索、新建、修改、导入、审核 |
| 企业梦境 | Dream Policy、Dream Run、Dream Summary | 逐级查看组织运行记录如何被蒸馏为企业记忆 |
| 做事流程 | Knowledge Workflow | 查看和维护 SOP 步骤；高级模式进入 FlowGram |
| 回答依据 | Answer Trace、Evidence Pointer | 查看回答用了哪些知识和证据 |

基础模式不显示解析 Provider、索引、内部 ID、节点类型和诊断信息。高级维护模式只对具备权限的服务商或指定高级维护人员显示。

### 4.1 企业梦境不是普通摘要

Dream 是 AgentAtlas 的组织记忆蒸馏机制。它按企业设计好的工作流，持续把员工、项目组、部门、事业部和公司各层级的运行记录，逐级整理成可检索、可追溯、可治理的企业记忆。

```text
员工运行记录与工作记忆
  → 项目组 Dream Summary
  → 部门 Dream Summary
  → 事业部 Dream Summary
  → 公司 Dream Summary
```

默认上卷规则是“下一级已脱敏 Dream Summary 优先”：上级 Dream 消费下级已经完成范围收敛、脱敏和可见性判断的摘要，而不是默认汇集所有员工原始记录。只有异常调查、风险追溯或具有明确授权的审计任务，才能通过 AgentNexus 获取短期 Step Grant，沿证据指针下钻到原始材料。

每次 Dream 运行至少完成：

1. 固定时间窗口、组织范围和输入快照。
2. 合并本层运行记录和已发布的下级 Dream Summary。
3. 在模型调用前执行字段级脱敏和可见性收敛。
4. 提取事实、主题、趋势、风险、待办和缺失输入。
5. 生成展示摘要、检索摘要和密封详情指针三层输出。
6. 保存输入血缘、策略版本、工作流版本、模型版本、证据指针和上级可消费关系。
7. 根据策略自动发布、等待确认或进入复核队列。

Dream 产物是某一时间窗口和策略版本的不可变运行结果。用户不能直接改写历史摘要；可以确认、驳回、添加批注、标记错误、修改后续策略、重新运行，或在有权限时追溯证据。这样历史判断和审计链不会被事后覆盖。

### 4.2 企业梦境的四个视图

- **梦境全景**：按员工、项目组、部门、事业部和公司展示层级关系，以及最近运行、覆盖率、新鲜度、缺失输入和风险数量。
- **梦境时间线**：按组织层级和时间窗口展示摘要、趋势、风险、待办及其状态；历史结果只读。
- **梦境工作流**：基础模式用人话向导配置范围、频率、输入、输出和是否需要确认；高级维护模式使用 FlowGram 编辑完整 Dream Workflow。
- **运行与追溯**：展示本次输入组织、下级摘要、策略和工作流版本、输出层、失败步骤、证据链与审计记录；敏感证据下钻必须再次授权。

“工作记忆”保留为员工或低层组织的输入/输出对象名称，不再作为整个 Dream 机制的顶层产品名称。

## 5. 常态首页

常态首页以“组织知识浏览”为主体：

- 顶部：AgentAtlas 品牌、四个工作面、仅图标的 Atlas Agent 按钮、用户信息。
- 顶部不放常驻搜索输入框；内容搜索位于“已有内容”工具栏，需要时展开。
- 左侧：当前用户有权维护的组织范围、最近修改、待审核数量。
- 中间：当前组织标题、运行状态、用途说明、四个任务快捷入口和已有知识列表。
- 四个快捷入口：添加资料、新建或修改知识、制作 SOP 流程、处理建议与审核。
- 列表行显示普通人能理解的名称、类型、更新时间、适用范围，以及明确的“查看/编辑/更新”操作。

页面第一眼必须回答：

1. 这里用于做什么。
2. 当前维护哪个组织范围。
3. 用户可以做哪几件事。
4. 从哪里开始。
5. 当前内容是否正常运行。

## 6. Atlas 助手

### 6.1 入口和布局

- Agent 按钮位于顶部用户信息左侧，仅显示专业线性图标，不显示容易换行的常驻文字。
- Agent 按钮、用户名和下拉箭头属于同一个不可换行操作组，固定同高并垂直居中。
- 桌面端点击后从右侧展开占位式抽屉，主内容区同步缩窄和重排，抽屉不得覆盖或隐藏正文。
- 抽屉关闭后页面恢复原宽度，顶部 Agent 按钮始终可见。
- 窄屏不使用覆盖正文的浮窗；进入独立的全屏助手视图，返回后恢复原页面和编辑上下文。

### 6.2 能力边界

Atlas 助手是 AgentAtlas 自己的 Knowledge Agent 交互入口，不是嵌入的小智 Claw。它可以：

- 根据当前组织范围添加或修改知识。
- 接收附件并创建资料解析任务。
- 生成 SOP 草稿。
- 解释变更、风险、影响范围和回答依据。
- 帮助低电脑能力用户进入正确维护任务。

权限检查发生在入口和每次具体操作上。对话不能绕过建议、编辑、发布和复核权限。

## 7. 知识查看与编辑

普通维护者使用“结构化说明书”模式，而不是直接面对流程节点：

- 左侧：内容目录，例如基本说明、开始前准备、处理步骤、需要的证据、完成标准。
- 中间：可阅读、可编辑的结构化内容。SOP 步骤使用编号卡片，上下按钮调整顺序。
- 右侧：适用组织范围、影响人数、参考资料和高级维护入口。
- 编辑内容自动保存为草稿，并明确显示真实保存状态。
- 每页只有一个主要操作：“下一步：检查并发布”。
- FlowGram 完整画布只通过“高级维护 → 查看流程图”进入。

普通编辑页不显示 Workflow JSON、节点注册名、Schema 字段、数据库 ID 或运行时配置。

## 8. 建议、风险与审批

### 8.1 权限不足的员工

没有编辑权限的员工可以提交纠错建议。建议包含问题说明、关联知识、可选证据和组织范围，但不会直接改变已发布知识。

### 8.2 风险判定

风险由确定性规则计算，至少考虑：

- 修改对象类型。
- 是否修改已发布 SOP 或工作流。
- 影响组织范围和人数。
- 是否影响权限、审批、执行时限、证据要求或外部动作。
- 企业为特定知识类型设置的最低风险级别。

企业管理员可以提高风险级别，普通用户和 Agent 不能临时降低风险。

### 8.3 发布规则

- 低风险：具备 `publish_low_risk` 权限的用户检查修改前后和影响范围后，可以单人确认发布。
- 高风险：必须由组织架构向上的另一位合格负责人复核。
- 发起人不能成为高风险修改的第二复核人。
- 系统优先寻找当前组织上一级中具有对应发布权限的负责人，并按组织架构继续向上。
- 找不到合格复核人时，进入企业知识管理员待分配队列，绝不自动发布。

### 8.4 检查页面

检查页面必须使用普通语言展示：

- 修改前和修改后。
- 风险级别及确定性判定原因。
- 生效后影响的组织、人数、员工 Agent 回答和 SOP 执行。
- 下一位复核人的姓名、职责和组织审批路径。
- 当前状态是草稿、待复核还是已发布。

高风险页面只有一个主要操作“提交给某负责人复核”；低风险页面对应“确认并发布”；无编辑权限页面对应“提交纠错建议”。

## 9. 身份、会话与登录

AgentAtlas 不保存或收集企业登录凭证。登录采用“无感会话 + 未登录整页跳转”：

```text
浏览器打开 AgentAtlas
  → GET /api/session
  → 已登录：直接进入或恢复原任务
  → 未登录：整页跳转 AgentNexus /authorize
  → AgentNexus 跳转企业统一身份认证
  → AgentAtlas /auth/callback
  → 设置 HttpOnly + Secure 会话 Cookie
  → 返回原页面
```

协议采用 OIDC Authorization Code + PKCE。禁止：

- 登录弹窗。
- AgentAtlas 自建账号密码表单。
- 浏览器 JavaScript 长期保存身份 token。
- 用户复制或粘贴 Access Ticket。

浏览器登录会话只证明“当前是谁”。每次读取证据、修改、发布或高级诊断时，AgentAtlas 后端仍依据会话身份向 AgentNexus 申请短期 Access Ticket/Step Grant。

当前 AgentNexus 已有 Access Ticket 和 Step Grant 基础，但没有完整的 Console OIDC/Browser Session 实现。实施本设计需要在 AgentNexus 增加公共 Browser Session、OIDC authorize/callback、会话查询和退出契约。

## 10. 权限模型

基础操作权限：

| 权限 | 含义 |
|---|---|
| `suggest` | 提交纠错建议 |
| `edit` | 创建和修改草稿 |
| `publish_low_risk` | 发布本组织低风险修改 |
| `approve_high_risk` | 复核下级组织高风险修改 |
| `workflow_edit` | 修改普通 SOP 步骤 |
| `workflow_advanced` | 进入 FlowGram 完整画布和节点配置 |
| `service_mode` | 查看解析、连接、版本、索引和诊断能力 |

每项权限同时绑定企业、组织范围、知识类型和操作类型。服务商进入高级模式后仍不能突破客户企业或授权组织范围；查看、修改、重建索引和诊断操作全部审计。

## 11. 共享 UI 架构

### 11.1 依赖方向

AgentAtlas 不直接引用 `xiaozhiclaw-runtime/ui-next/src`，也不继续维护镜像 UI 包。

```text
xiaozhiclaw-runtime/ui-next ─┐
                            ├→ @xiaozhiclaw/runtime-ui
agentatlas-console ─────────┤
agentnexus-console ─────────┘
```

在 `xiaozhiclaw-runtime/packages/runtime-ui` 建立独立构建、测试和发布的 npm 包 `@xiaozhiclaw/runtime-ui`。它初期留在 `xiaozhiclaw-runtime` 仓库，不单独创建 Git 仓库。小智自身必须先消费该包，确保共享组件不是只为外部产品维护。

### 11.2 共享内容

- DesignProvider、主题、字体、密度和全部设计 tokens。
- Button、Input、Menu、Dialog、Sheet 等基础组件。
- `PromptInput`。
- MessageList、消息展示原语、ApprovalCard、AttachmentCard。
- 不覆盖正文的 DockedPanel。
- Lucide React 图标规范。

### 11.3 不进入共享包

- 小智 stores 和 gateway WebSocket。
- 桌面原生桥和会话状态。
- 模型选择、账号状态、语音、目标模式等小智业务能力。
- AgentAtlas 知识对象、API 和权限业务逻辑。

`PromptInput` 已经是由 `value/onChange/onSubmit` 驱动的纯受控组件，可以直接迁入共享包。小智完整 `InputBox` 与 stores、事件总线、模型和语音能力耦合，不整体共享；应把纯 UI 外壳和业务适配分开。

AgentAtlas 通过自己的适配层向共享聊天 UI 传入 `messages`、`onSend`、附件回调和当前知识范围。迁移完成后删除 AgentAtlas 当前镜像性质的 `packages/claw-runtime-ui`。

### 11.4 视觉规范

视觉严格遵循 `xiaozhiclaw-runtime/ui-next/DESIGN.md`：

- 暖纸背景 `#F9F8F0`。
- 墨黑 `#131314`。
- 陶土 `#D97757` 是唯一主动彩色；主按钮默认墨底，hover/active 使用陶土。
- 使用低饱和土系功能色：陶土赭、橄榄、金赭、暗青。
- 页面卡使用 Warm Paper × Glass 半透明玻璃、高光内描边和分层阴影。
- 页面级中文标题采用思源宋体方向；正文使用 Hanken Grotesk/Noto Sans SC；元数据使用 JetBrains Mono。
- sidebar active 使用陶土渐变和左强调线。
- 图标统一使用 Lucide React，保持一致尺寸、stroke、round cap/join。
- 禁止 Emoji 充当产品功能图标，禁止硬编码系统蓝绿红，禁止复制一套近似 tokens。

### 11.5 现有 Console 全量重构

本设计不是在现有 Console 旁边增加一套新页面。现有前端全部工作面必须迁入已确认的信息架构、共享组件和 Warm Paper × Glass 家族设计；迁移完成后不得长期保留新旧两套壳、tokens、聊天组件或视觉实现。

| 现有组件 | 重构后的定位 |
|---|---|
| `AgentAtlasDashboard` | 新的统一产品壳：四个工作面、组织范围、权限感知导航、顶部 Agent 按钮和用户会话 |
| `KnowledgeMap` | “企业知识”中的只读组织知识图视图，不作为低电脑能力用户的默认首页 |
| `DreamTimeline` | 拆分为“梦境全景”和“梦境时间线”，展示逐级蒸馏关系、摘要、趋势、风险、待办、覆盖率和新鲜度 |
| `DreamPolicyPanel` | 重构为“梦境工作流”基础向导；cron、原始规则、FlowGram 和运行诊断只在高级维护模式显示 |
| `WorkflowStudio` | 拆分为工作流列表、结构化 SOP 基础编辑器和 FlowGram 高级编辑器 |
| `AtlasWorkflowCanvas` | 保留为 FlowGram 封装边界，重做节点、工具栏、状态和空页面的家族视觉 |
| `AnswerTraceGraph` | 重构为“回答依据”：先显示人话摘要和来源清单，需要时展开 React Flow 只读证据图 |
| `AtlasAgentPanel` | 删除独立实现，改为顶部 Agent 按钮 + `@xiaozhiclaw/runtime-ui` 的 DockedPanel、消息流和 PromptInput |
| `packages/claw-runtime-ui` | 完成共享包迁移后删除，改用 `@xiaozhiclaw/runtime-ui` |

重构要求：

- 所有页面使用共享 tokens、字体、按钮、输入、菜单、Dialog、消息流和图标规范。
- 页面不再要求用户粘贴管理员票据；身份来自 AgentNexus Browser Session。
- 基础模式不出现内部 ID、英文节点类型、cron、原始 JSON、索引名和 Provider 配置。
- 空状态、加载、保存失败、无权限和发布结果必须采用同一套家族反馈组件。
- 旧页面只有在对应新页面通过功能与视觉验收后才能删除；迁移期通过路由或 feature flag 切换，不在同一页面混用两种设计系统。

### 11.6 Dream 运行时与 Console 迁移

现有 Dream 代码已经具备策略版本、定时运行、幂等运行号、运行输入、三层摘要、对象存储和时间线基础，但当前 Runner 只读取输出组织的直属成员 `work_brief`，并由硬编码聚合器直接生成摘要。现有 `dream.aggregate` 工作流节点尚未成为定时运行的权威执行路径，策略创建接口也把“创建草稿”和“发布”合并成了一次操作。

迁移后必须形成单一权威 Dream 执行模型：

```text
Published DreamPolicyVersion
  → pinned DreamWorkflowVersion
  → immutable DreamRun(input snapshot + visibility snapshot)
  → DreamSummary(display + retrieval + sealed pointer)
  → parent organization input or governed evidence drill-down
```

后端需要补齐：

- 按组织树深度调度，只有下级窗口完成或明确标记缺失后，上级窗口才可运行。
- `work_brief`、项目记录、SOP 变更、Agent 回答、外部证据、已完成任务、风险事件和下级 Dream Summary 的输入解析器。
- 策略草稿、检查、发布、停用、手工重跑和回填生命周期；发布与高风险变更进入统一治理流程。
- 绑定并执行固定版本 Dream Workflow，移除 Scheduler 到硬编码聚合器的旁路。
- 结构化保存 trends、todos、coverage、missing inputs、parent/child run lineage、visibility snapshot、previous state 和失败步骤。
- 企业时区、最大重试次数、并发窗口、幂等重跑和部分输入失败语义。
- Dream 专用公共 API Schema；Console 和 Enterprise 集成不得依赖 Go `internal/dream.Policy`。
- 每次运行及证据下钻写入 Answer Trace 和审计；AgentNexus 不可用或审计失败时敏感读取失败闭合。

前端不再把 Dream 做成平铺摘要列表或原始 cron/正则表单。基础模式优先解释“正在整理哪个组织、用了哪些下级记忆、当前结果是否完整、接下来谁需要处理”；高级维护模式才展示节点、原始规则、调度表达式和诊断数据。

### 11.7 现有工作流迁移与双编辑器策略

现有 `WorkflowStudio` 和 `AtlasWorkflowCanvas` 保留其公共 Workflow Schema 与 FlowGram 封装价值，但不能原样作为最终普通用户编辑体验。

当前实现的迁移基线：

- `WorkflowStudio` 主要通过节点目录、线性 `appendNode/removeNode` 和步骤标签修改 React 状态。
- `AtlasWorkflowCanvas` 已有 `onExport` 能力，但当前 `WorkflowStudio` 未接入，画布内拖动和连线修改没有进入保存状态。
- 当前保存调用会创建新工作流，缺少加载现有草稿、更新草稿、并发版本和差异接口。
- 当前发布按钮直接调用发布 API，尚未进入本设计的风险检查和向上审批流程。

最终提供两个编辑器，但只使用一个权威 `AtlasWorkflow` 文档：

```text
普通维护者
  → SOPStructuredEditor
  → 人话步骤、顺序、证据、确认点
  → 编译为 AtlasWorkflow Schema

服务商或高级维护者
  → AdvancedWorkflowStudio
  → FlowGram 节点、边、条件和配置
  → 导出为同一个 AtlasWorkflow Schema
```

兼容与防丢失规则：

- 线性 SOP 可以在结构化基础编辑器中完整修改。
- 存在分支、条件或基础编辑器不支持节点的工作流，在基础模式只显示结构化摘要和只读步骤。
- 基础编辑器不得把高级工作流静默压平为线性流程。
- 进入 FlowGram 前必须具备 `workflow_advanced` 权限。
- 实现 `AtlasWorkflow → WorkflowJSON` 与 `WorkflowJSON → AtlasWorkflow` 双向转换。
- round-trip 测试必须证明 `config`、edge condition、确认点、风险级别、变量和输入输出 Schema 不丢失。
- 编辑已发布版本时创建新草稿版本，不修改历史版本。
- 工作流发布统一进入“检查并发布/提交复核”页面，移除基础模式中的直接发布动作。

后端需要补充工作流列表、详情读取、基于版本创建草稿、更新草稿、乐观并发、草稿差异、风险检查、审批和幂等发布契约。现有已发布工作流不进行破坏性重写；首次编辑时从指定已发布版本创建新草稿。

## 12. 端到端数据流

### 12.1 直接维护

```text
用户打开 AgentAtlas
  → AgentNexus 会话/SSO 验证
  → 返回身份、组织范围和操作权限
  → AgentAtlas 展示授权知识
  → 用户创建草稿
  → 按风险策略直接发布或向上复核
  → 更新检索和员工 Agent 可用知识
  → 写入版本与审计
```

### 12.2 小智纠错建议

```text
员工在小智提出错误
  → AgentNexus 校验 suggest 权限
  → AgentAtlas 创建建议
  → 分配给对应组织知识负责人
  → 负责人修改或拒绝
  → 发布后通知建议人结果
```

### 12.3 高风险发布

```text
发起人提交
  → 确定性规则判定高风险
  → AgentNexus 读取组织架构与发布权限
  → 排除发起人
  → 找到向上第一位合格复核人
  → 复核通过
  → 原子发布新版本
  → 更新索引
  → 审计追加成功后对外生效
```

## 13. 失败处理

- 会话过期：保存当前任务位置，整页重新登录，返回后恢复任务。
- 权限不足：允许提交建议，并说明建议将交给谁处理，不显示只有错误码的死路页面。
- 并发修改：禁止静默覆盖，展示用户草稿与最新版本差异，并允许重新应用修改。
- 自动保存失败：固定提示“尚未保存”，保留当前内存草稿并允许重试；不能显示虚假成功。
- 发布超时：草稿保持不变；使用幂等键查询最终状态，禁止重复发布。
- 找不到复核人：进入企业知识管理员待分配队列。
- AgentNexus 不可用：身份、证据读取、审批和发布全部失败闭合。
- 审计追加失败：修改不能进入已发布状态。
- 多文件导入部分失败：成功文件继续处理；失败文件逐个显示原因和重试入口。

敏感知识草稿不长期写入浏览器 `localStorage`。网络中断时只保留当前页面内存；离开页面前必须明确提示尚未保存。

## 14. 可访问性与低电脑能力原则

- 每页最多一个主要操作。
- 所有页面明确显示当前位置、用途、当前状态和下一步。
- 基础模式只使用业务语言，不暴露技术术语。
- 危险或高风险操作必须先显示影响范围。
- Atlas 助手不能覆盖正文。
- 支持键盘操作、清晰焦点状态和 WCAG AA 文字对比。
- 200% 缩放及 1280×720 分辨率下仍能完成核心任务。
- 空状态给出下一步，不只显示“暂无数据”。
- 错误状态解释发生了什么、数据是否保存、用户接下来能做什么。

## 15. 验收标准

### 15.1 零培训可用性测试

邀请未使用过 AgentAtlas、电脑能力较弱的企业人员，在不培训的情况下完成：

1. 找到研发一部的知识。
2. 上传一份资料。
3. 修改一条错误知识。
4. 调整一个 SOP 步骤。
5. 提交一次高风险修改给上级复核。
6. 提交一条无编辑权限的纠错建议。

通过标准：

- 至少 80% 的首次用户能在五分钟内独立完成每项任务。
- 100% 用户能说清修改处于草稿、待复核还是已发布状态。
- 100% 用户能找到当前组织范围和下一步操作。
- 没有用户需要复制 token、输入内部 ID、理解 JSON、cron、索引或节点类型。
- 无权限用户不能直接发布。
- 高风险发起人不能复核自己的修改。

### 15.2 工程验证

- `@xiaozhiclaw/runtime-ui` 在小智和 AgentAtlas 中运行同一套组件测试和视觉回归。
- AgentAtlas Console 不再导入 `@agentatlas/claw-runtime-ui`，仓库中不存在复制的 tokens 或聊天组件源码。
- 首页、梦境全景、梦境时间线、梦境工作流、基础 SOP、FlowGram 高级画布和回答依据全部通过家族设计视觉回归。
- Dream 自底向上运行顺序、下级脱敏摘要优先、缺失输入、授权证据下钻、不可变运行结果和重跑血缘均有集成测试。
- `AtlasWorkflow ↔ WorkflowJSON` 双向 round-trip 覆盖全部节点类型、条件、配置和 Schema 字段。
- 登录整页跳转、回调、原页面恢复和退出 E2E。
- 权限矩阵、风险判定、组织向上审批和无复核人回退测试。
- 草稿自动保存、并发冲突、幂等发布和审计失败闭合测试。
- 使用真实 AgentNexus、AgentAtlas、PostgreSQL、OpenSearch 和对象存储运行集成测试。
- 1280×720、200% 缩放、键盘导航和抽屉展开重排测试。

## 16. 实施拆分边界

本设计需要按依赖顺序拆成四个实施工作流：

1. `xiaozhiclaw-runtime`：建立 `@xiaozhiclaw/runtime-ui` 并让小智自身消费。
2. AgentNexus：新增 Browser Session、OIDC/PKCE、组织范围权限和向上审批公共契约。
3. AgentAtlas 后端：Dream 分层蒸馏运行时，以及建议、草稿、风险判定、版本、复核和幂等发布能力。
4. AgentAtlas Console：重构现有产品壳和全部工作面，交付首页、知识编辑、梦境全景、梦境时间线、梦境工作流、运行与追溯、双工作流编辑器、回答依据、检查发布、Atlas 助手和高级维护模式。

四个工作流不得通过跨仓深层源码引用连接，只能依赖已发布包和公共 API 契约。
